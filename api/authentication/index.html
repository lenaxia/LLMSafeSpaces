    Authentication¶

    LLMSafeSpaces uses two complementary authentication mechanisms:

    1. JWTs issued by /auth/register and /auth/login, delivered via an HttpOnly cookie or a Bearer header.
    2. API keys with the lsp_ prefix, generated by an authenticated user and suitable for long-lived programmatic access.

    Both are accepted by the same AuthMiddleware and presented the same way: Authorization: Bearer <token>. The middleware inspects the token, recognizes whether it is a JWT or an API key (by the lsp_ prefix), validates it, and stashes the resolved user ID on the request context.

    When to use which¶

    Mechanism Best for Lifetime Revocable
    JWT (cookie) Browser sessions, the frontend SPA Hours (default 24h, configurable) Per-token (jti) via logout/revocation
    JWT (Bearer header) Short-lived CLI/scripts that log in As above As above
    API key (lsp_…) CI, long-lived integrations, the SDKs, MCP clients Until explicitly revoked Per-key via DELETE /auth/api-keys/:id

    API keys are the right default for automation

    JWTs require a login round-trip and expire. API keys are stable, individually revocable, and SHA-256 hashed at rest — use them for anything that runs unattended.

    JWT¶

    JWTs are signed with golang-jwt/v5 using the configured auth.jwtSecret. They are issued by Register and Login and returned in the JSON body ({token, user}) and set as the lsp_session HttpOnly cookie.

    Claims¶

    Claim Meaning
    sub User ID
    jti Unique token ID — used for per-token revocation
    exp Expiry (default 24h, configurable via auth.tokenDuration)
    iat Issued-at timestamp
    iss Issuer
    aud Audience

    The jti is what makes per-token revocation possible: logout writes the token's jti to a revocation set, and subsequent validations reject any token whose jti is present.

    Delivery: cookie vs header¶

    The lsp_session cookie is HttpOnly, Secure, and SameSite=Lax (so the SSO callback redirect works). The API accepts the JWT from either source:

    1. Authorization: Bearer <jwt> header (priority)
    2. lsp_session cookie

    The cookie is the browser path; the header is the programmatic path. Both end up at the same validator.

    Token revocation (jti-based)¶

    POST /auth/logout revokes the current JWT. The handler:

    1. Extracts the token from the header or cookie.
    2. Skips API keys (they have their own lifecycle via /auth/api-keys/:id).
    3. Calls RevokeToken, which adds the jti to the revocation cache.
    4. Clears the cookie and returns 204 regardless of whether revocation succeeded.

    Logout always succeeds from the user's perspective

    Revocation errors are logged at WARN but do not propagate. The cookie is cleared and 204 is returned even if the revocation write failed — a user-facing logout failure is worse than a best-effort revocation miss.

    The token extractor is header-only by default: query-parameter and cookie extraction are disabled. This avoids leaking tokens into access logs and referrers. The lsp_session cookie is read only by the explicit logout handler, not by the general token extractor.

    API keys¶

    API keys are the stable, long-lived alternative to JWTs.

    Generation¶

    • Prefix: lsp_ (configurable via auth.apiKeyPrefix).
    • Entropy: 32 random bytes from crypto/rand, base64-encoded.
    • At rest: SHA-256 hashed. The plaintext key is returned only at creation time and is never recoverable.
    • Listing: GET /auth/api-keys strips the secret — it returns metadata only.
    • Revocation: DELETE /auth/api-keys/:id removes the key.
    # Create an API key (the plaintext is returned exactly once)
curl -sX POST "$API/api/v1/auth/api-keys" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"ci-runner"}'
# → { "id": "...", "name":"ci-runner", "key":"lsp_...", "createdAt":"..." }

# List keys (secret stripped)
curl -s "$API/api/v1/auth/api-keys" -H "Authorization: Bearer $TOKEN"
# → [ { "id":"...", "name":"ci-runner", "createdAt":"..." } ]

# Revoke a key
curl -X DELETE "$API/api/v1/auth/api-keys/$KEY_ID" -H "Authorization: Bearer $TOKEN"

    Validation¶

    API keys are recognized by their lsp_ prefix. On every request, the middleware:

    1. Extracts the bearer token.
    2. Checks for the lsp_ prefix.
    3. If present, SHA-256 hashes it and looks up the hash. If it matches, the user ID is loaded onto the context.
    4. Otherwise, treats it as a JWT and validates signature + expiry + revocation.

    API keys do not carry a jti and are not added to the JWT revocation cache on logout — they have their own delete endpoint.

    Password handling¶

    • Hashing: bcrypt at cost 12.
    • Never in responses: User.PasswordHash is tagged json:"-" and verified absent in e2e tests.
    • Email enumeration prevention: duplicate email, wrong password, nonexistent user, and inactive user all return identical generic errors. There is no way to distinguish "user exists" from "wrong password" via the API.
    • Binding errors sanitized: JSON binding failures return {"error":"invalid request body"} rather than echoing internal struct details.

    Rate limiting¶

    Rate limiting is Redis-backed (with an in-memory token-bucket fallback) and wired into the global middleware stack.

    Config key Env var Default Purpose
    rateLimiting.enabled LLMSAFESPACES_RATELIMITING_ENABLED false Enable rate limiting
    rateLimiting.defaultLimit LLMSAFESPACES_RATELIMITING_DEFAULTLIMIT 100 Requests per window
    rateLimiting.defaultWindow LLMSAFESPACES_RATELIMITING_DEFAULTWINDOW 1m Window duration
    rateLimiting.burstSize LLMSAFESPACES_RATELIMITING_BURSTSIZE 20 Burst allowance

    The auth group has its own stricter limiter: 20 requests/minute, burst 5, and a 1 MiB body limit (http.MaxBytesReader) on every auth endpoint.

    Long-lived connections are exempt so reconnects after network drops don't trigger 429s:

    • /events (user-scoped SSE)
    • /session-events (session-scoped SSE)

    When no API key is in context, the limiter falls back to c.ClientIP() as the rate-limit identity.

    Account lockout¶

    Optional brute-force protection on login. When enabled, repeated failed logins lock the account for a configured duration.

    Config key Env var Default Purpose
    auth.lockoutEnabled LLMSAFESPACES_AUTH_LOCKOUTENABLED false Enable account lockout
    auth.lockoutAttempts LLMSAFESPACES_AUTH_LOCKOUTATTEMPTS 0 Failed attempts before lockout (e.g. 5)
    auth.lockoutDuration LLMSAFESPACES_AUTH_LOCKOUTDURATION 0 Lockout duration (e.g. 15m)

    JWT cache hardening¶

    Validated JWTs are cached in Redis to avoid re-verifying the signature on every request. The cache key is the MD5 hash of the raw JWT, never the JWT itself — so a Redis compromise does not expose live bearer tokens. See hashToken() in the auth service.

    Cloudflare Turnstile CAPTCHA on register¶

    POST /auth/register can be gated behind Cloudflare Turnstile. The feature is disabled by default and feature-flagged via chart.turnstile.enabled. When enabled:

    • The frontend renders the Turnstile widget on the register page.
    • The client sends the Turnstile token in the cf-turnstile-response header.
    • The API middleware validates the token against Cloudflare's siteverify endpoint before invoking Register.

    The middleware fails closed on every failure mode with 401 {"error":"turnstile_failed","reason":<code>}:

    Failure reason
    Missing token missing_token
    siteverify HTTP error / timeout verify_unavailable
    siteverify non-200 verify_unavailable
    siteverify success:false rejected (Cloudflare error codes in detail)

    A startup guard refuses to boot the API with Turnstile.Enabled=true and an empty SecretKey, so the "no secret configured" state is unreachable in practice.

    What Turnstile does NOT gate

    • POST /auth/login — credential-stuffing is already covered by Cloudflare rate-limit + account lockout.
    • POST /auth/lookup, /sso/domains — these are enumeration-safe by design; CAPTCHA friction would degrade legitimate login discovery.

    See the operator guide for provisioning (Cloudflare dashboard, AWS Secrets Manager, CDK, Flux reconciliation).

    Per-workspace ownership¶

    Authentication establishes who you are. Authorization on workspace routes is enforced by a single ownership gate: WorkspaceAccessMiddleware. Every /:id workspace route (including all proxied session routes) inherits it. It verifies that the caller is the workspace's owner (a user, or a member/admin of the owning org) before the handler runs.

    List and Create have no :id and bypass the middleware — List is scoped per-user inside the service, and Create has no target workspace yet.

    See Concepts for the WorkspaceOwner{UserID, OrgID} ownership model.

    Security controls summary¶

    Control Implementation
    Password hashing bcrypt cost 12
    Email enumeration prevention Identical generic errors for all auth failures
    Password never in response json:"-" on User.PasswordHash; verified in e2e
    API key secret stripped on list ListAPIKeys zeroes the Key field before return
    API key secret returned only on creation CreateAPIKey returns the full key once
    Body size limits http.MaxBytesReader (1 MiB) on all auth endpoints
    Sanitized binding errors Generic "invalid request body" on bind failure
    No internal error leakage Service errors return generic messages; details logged server-side
    JWT includes jti Enables per-token revocation
    API keys use crypto/rand 32-byte random keys with lsp_ prefix
    JWT cache keys hashed MD5 before Redis storage
    Token extraction header-only Query param and cookie extraction disabled by default
    Rate limiter wired globally Redis-backed + in-memory token bucket
    Wrong HTTP method rejection Only POST on register/login returns 404 for GET
    Turnstile CAPTCHA on register Optional, feature-flagged, fail-closed
