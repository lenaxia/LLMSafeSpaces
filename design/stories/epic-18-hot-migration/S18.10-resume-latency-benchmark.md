# S18.10: Workspace Startup Latency Benchmarking

**Epic:** 18 — Hot Migration & RWX Storage
**Status:** Planning
**Priority:** High — prerequisite for all cold-start optimisation work
**Depends on:** None (runs against current cluster)
**Estimated Effort:** 4 points

---

## Objective

Measure and instrument two distinct workspace startup paths:

1. **Resume** (Suspended → Active): observed up to 2 minutes — 4–40× the "~3s" figure
   cited in the architecture doc.
2. **Create** (first-ever creation, Pending → Active): never benchmarked; involves PVC
   provisioning and optionally the `workspace-setup` init container (package installs,
   initScript). Likely slower than resume for non-trivial workspaces.

Both paths share some gates but diverge significantly. They have different optimization
levers and must be measured separately. This story instruments every gate in both paths
and produces histogram baselines that all subsequent stories can regress against.

---

## Background: Gate Maps

### Resume path gates (Resuming → Active)

A workspace in Suspended phase has an existing bound PVC. Resume creates a new pod
against the pre-existing PVC.

| Gate | Description | Worst-case contribution |
|------|-------------|------------------------|
| **R1** Pod scheduling | Kubernetes assigns node, pulls image (cached = fast), runs init containers (credential copy only — no package install) | 2–10s (cached image) or 30–60s (cache miss) |
| **R2** `InitialDelaySeconds` | Kubelet hard-waits before first readiness probe | 10s hard floor (`pod_builder.go:103`) |
| **R3** opencode startup | `opencode serve` loads Node.js, opens existing SQLite WAL on PVC, loads provider SDKs | 5–20s (warm SQLite state on PVC — faster than create) |
| **R4** agentd healthz cache | `refreshIsHealthyLoop` polls `/global/health` every 5s; cache starts `Healthy=false` | 0–10s per miss |
| **R5** `providers_connected` | `readyzHandler` requires `len(connected) > 0`; upstream LLM API round-trip | **0–120s (suspected dominant gate)** |
| **R6** kubelet→controller | Kubelet marks pod Ready → controller reconcile observes Running+PodIP | 0–5s (`requeueCreating=5s`) |

### Create path gates (Pending → Active)

A workspace being created for the first time has no PVC yet.

| Gate | Description | Worst-case contribution |
|------|-------------|------------------------|
| **C1** PVC provisioning | StorageClass dynamic provisioner creates volume and binds PVC | 2–30s (EFS access point creation) or 1–5s (local) |
| **C2** Pod scheduling | Node assignment + image pull (likely cold on new clusters) | 2–60s |
| **C3** `workspace-setup` init container | Runs if `spec.packages` or `spec.initScript` set: pip/npm/go install | **0s (no packages) to minutes (large package set)** |
| **C4** `credential-setup` init container | Copies secret files to tmpfs sandbox-cfg | <1s |
| **C5** `InitialDelaySeconds` | Same 10s hard floor | 10s |
| **C6** opencode startup | `opencode serve` initialises fresh SQLite DB on new PVC | 5–30s (cold DB — potentially slower than resume) |
| **C7** agentd healthz cache | Same as R4 | 0–10s |
| **C8** `providers_connected` | Same as R5 | 0–120s |
| **C9** kubelet→controller | Same as R6 | 0–5s |

### Key differences

| | Resume | Create |
|---|--------|--------|
| PVC state | Pre-existing, bound | Must be provisioned |
| SQLite state | Warm (previous session history on PVC) | Cold (fresh init) |
| Init containers | credential-setup only (~1s) | workspace-setup (variable) + credential-setup |
| Image pull | Usually cached | May be cold |
| Optimisation levers | Gate R5 (decouple readyz), probe tuning, gVisor checkpoint/restore | Gate C1 (PVC pre-provisioning), Gate C3 (package caching), Gate C5–C8 (same as resume) |

## Scope

### Part 1 — Prometheus instrumentation (code changes)

**1a. New histograms: create and resume duration**

Add to `controller/internal/metrics/metrics.go`:

```go
WorkspaceCreateDurationSeconds = prometheus.NewHistogramVec(
    prometheus.HistogramOpts{
        Name:    "llmsafespace_workspace_create_duration_seconds",
        Help:    "Wall-clock time from Pending phase entry to phase=Active (first creation)",
        Buckets: []float64{1, 3, 5, 10, 15, 20, 30, 45, 60, 90, 120, 180, 300},
    },
    []string{"has_packages", "has_init_script"},
)

WorkspaceResumeDurationSeconds = prometheus.NewHistogramVec(
    prometheus.HistogramOpts{
        Name:    "llmsafespace_workspace_resume_duration_seconds",
        Help:    "Wall-clock time from Resuming phase entry to phase=Active",
        Buckets: []float64{1, 3, 5, 10, 15, 20, 30, 45, 60, 90, 120, 180},
    },
    []string{"resume_type"}, // "first_resume" | "subsequent_resume"
)
```

**Create labels:**
- `has_packages`: `"true"` if `len(spec.packages) > 0`, else `"false"` — isolates
  workspace-setup init container cost
- `has_init_script`: `"true"` if `spec.initScript != ""`, else `"false"`

**Resume labels:**
- `resume_type=first_resume`: `RestartCount == 0` (SQLite DB is cold on PVC)
- `resume_type=subsequent_resume`: `RestartCount > 0` (warm SQLite)

**1b. New status fields for timestamp anchors**

Add to `WorkspaceStatus` in `pkg/apis/llmsafespace/v1/workspace_types.go`:

```go
// PendingAt is set when the workspace first enters Pending phase.
// Used to compute create duration. Cleared on first Active transition.
PendingAt *metav1.Time `json:"pendingAt,omitempty"`

// ResumedAt is set when the workspace enters Resuming phase.
// Cleared when the workspace becomes Active. Used to compute resume duration.
ResumedAt *metav1.Time `json:"resumedAt,omitempty"`
```

Set `PendingAt` in `handlePending` on first entry (when `PendingAt == nil`):
```go
if workspace.Status.PendingAt == nil {
    now := metav1.Now()
    workspace.Status.PendingAt = &now
}
```

Set `ResumedAt` in `handleResuming`:
```go
now := metav1.Now()
workspace.Status.ResumedAt = &now
```

Record and clear both in `phase_creating.go` on Creating → Active transition:
```go
// Create path
if workspace.Status.PendingAt != nil && workspace.Status.ResumedAt == nil {
    elapsed := time.Since(workspace.Status.PendingAt.Time)
    hasPackages := strconv.FormatBool(len(workspace.Spec.Packages) > 0)
    hasInit := strconv.FormatBool(workspace.Spec.InitScript != "")
    metrics.WorkspaceCreateDurationSeconds.
        WithLabelValues(hasPackages, hasInit).
        Observe(elapsed.Seconds())
    workspace.Status.PendingAt = nil
}

// Resume path
if workspace.Status.ResumedAt != nil {
    elapsed := time.Since(workspace.Status.ResumedAt.Time)
    resumeType := "subsequent_resume"
    if workspace.Status.RestartCount == 0 {
        resumeType = "first_resume"
    }
    metrics.WorkspaceResumeDurationSeconds.
        WithLabelValues(resumeType).
        Observe(elapsed.Seconds())
    workspace.Status.ResumedAt = nil
}
```

**1c. Fix bucket range for `WorkspaceStartupDurationSeconds`**

Current max bucket is 60s. Add 90, 120, 180, 300 to cover observed range.

**1d. Per-gate agentd histograms**

Add to `cmd/workspace-agentd/healthz_cache.go`:

```go
var agentdGateDurationSeconds = prometheus.NewHistogramVec(
    prometheus.HistogramOpts{
        Name:    "llmsafespace_agentd_gate_duration_seconds",
        Help:    "Time from agentd boot to each startup gate (both create and resume paths)",
        Buckets: []float64{1, 2, 5, 10, 15, 20, 30, 45, 60, 90, 120},
    },
    []string{"gate"},
)
```

Gates (recorded once per boot, on first transition):
- `gate=opencode_up` — `time.Since(boot)` on first `Healthy=true`
- `gate=providers_connected` — on first `len(connected) > 0`
- `gate=readyz_first_200` — on first self-probe of `/v1/readyz` returning 200

These gates apply equally to both create and resume paths — agentd cannot distinguish
them, nor does it need to. The controller-side histograms provide the path label.

**1e. Register agentd `/metrics` endpoint**

Add to admin mux in `cmd/workspace-agentd/main.go`:

```go
adminMux.Handle("/metrics", promhttp.Handler())
```

Enables Prometheus scraping of per-pod agentd metrics from the cluster scrape config.

### Part 2 — Live benchmark scripts

Both scripts use only `kubectl`, `curl`, and `jq`. No new tooling required.

**`hack/benchmark-resume.sh`** — resume latency

```
Usage: ./hack/benchmark-resume.sh <workspace-id> [--iterations N] [--fresh-node]

Steps per iteration:
  1. POST /workspaces/:id/suspend → wait for phase=Suspended
  2. Record t0
  3. POST /workspaces/:id/activate
  4. Poll /workspaces/:id/status every 500ms → record t_creating, t_active
  5. kubectl get pod -w → record t_pod_running (Phase=Running)
  6. Port-forward agentd admin port → poll /v1/readyz every 500ms → record t_readyz_200
  7. POST proxy probe message → record t_proxy_ok (first 2xx)
  8. Print gate table; collect across N iterations for p50/p90/p99

Gates measured:
  API → Creating              (controller reconcile latency)
  Creating → Pod Running      (scheduling + init containers)
  Pod Running → readyz 200    (opencode startup + provider connect — expected dominant)
  readyz 200 → phase Active   (kubelet propagation + controller reconcile)
  phase Active → proxy ok     (proxy route)
```

**`hack/benchmark-create.sh`** — first-create latency

```
Usage: ./hack/benchmark-create.sh [--runtime base] [--packages "numpy pandas"] \
                                   [--init-script path/to/script.sh] \
                                   [--iterations N] [--fresh-node]

Steps per iteration:
  1. POST /workspaces (minimal spec, or with packages/initScript per flags)
  2. Record t0 = workspace Create call time
  3. Poll /workspaces/:id/status every 500ms → record:
       t_pending_to_creating: phase transitions from Pending to Creating
       t_active:              phase=Active
  4. kubectl get pod/:name -w → record:
       t_pod_created:         pod object appears in API
       t_init_complete:       all init containers Completed
       t_pod_running:         pod Phase=Running
  5. Port-forward agentd admin port → poll /v1/readyz → record t_readyz_200
  6. POST proxy probe → record t_proxy_ok
  7. DELETE /workspaces/:id (cleanup)
  8. Print gate table; collect for p50/p90/p99

Gates measured:
  API → PVC Bound             (PVC provisioning — storage class dependent)
  PVC Bound → Pod Created     (controller reconcile + pod schedule)
  Pod Created → Init Complete (workspace-setup + credential-setup init containers)
  Init Complete → Pod Running (main container start)
  Pod Running → readyz 200    (opencode startup + provider connect)
  readyz 200 → phase Active   (kubelet + controller)
  phase Active → proxy ok     (proxy route)

Run variants:
  --packages "":              baseline (no packages — workspace-setup init skipped)
  --packages "numpy pandas":  typical Python workspace
  --init-script ./setup.sh:   custom init
  --fresh-node:               cordon all other nodes to force cold image pull
```

**Expected output format (both scripts):**

```
=== Create Benchmark: runtime=base packages=none (run 1/5) ===

Gate                            Duration    Cumulative
----                            --------    ----------
API → PVC Bound                 3.1s        3.1s
PVC Bound → Pod Created         1.2s        4.3s
Pod Created → Init Complete     0.8s        5.1s
Init Complete → Pod Running     0.3s        5.4s
Pod Running → readyz 200        97.2s       102.6s  ← PRIMARY SUSPECT
readyz 200 → phase Active       3.3s        105.9s
phase Active → proxy ok         0.1s        106.0s

=== Summary (5 runs, no packages) ===
Gate                            p50     p90     p99
PVC provisioning                2.8s    5.1s    8.4s
Init containers                 0.7s    1.2s    2.1s
Pod Running → readyz 200        89s     108s    126s
Total (API → proxy ok)          98s     116s    139s
```

### Part 3 — Provider sub-breakdown logging

The `Pod Running → readyz 200` interval is opaque to the benchmark script because it
happens inside agentd. Structured log lines make it observable without a Prometheus
scrape:

In `refreshIsHealthyLoop`, log at `Info` level on first occurrence per boot:

```
[agentd] gate=opencode_up elapsed=8.3s
[agentd] gate=providers_connected elapsed=97.1s
[agentd] gate=readyz_first_200 elapsed=97.2s
```

If `opencode_up=8s` and `providers_connected=97s`, the 89s gap is entirely the
provider API check → confirms S18.11 is the right fix.

If `opencode_up=25s` and `providers_connected=26s`, the gap is in opencode startup
itself (Node.js load, SQLite init) → different optimization target.

- Unit: `handleResuming` sets `ResumedAt`
- Unit: `handleCreating` on Creating→Active records `WorkspaceCreateDurationSeconds` (when `PendingAt` set, `ResumedAt` nil), or `WorkspaceResumeDurationSeconds` (when `ResumedAt` set), and clears the anchor field

- Unit: gate recorder is a no-op if opencode never becomes healthy (no credentials configured)
- Integration: full Pending→Active cycle on envtest → `WorkspaceCreateDurationSeconds` observed
- Integration: suspend → resume cycle on envtest → `WorkspaceResumeDurationSeconds` observed

## Non-Goals

- This story does NOT fix the latency. It only measures it.
- No UI changes.
- No SLO enforcement. Metrics are for analysis and regression detection only.
- Package install benchmarking (deep dive into workspace-setup init container timing)
  is out of scope — the `has_packages` label on `WorkspaceCreateDurationSeconds` is
  sufficient to flag the issue; a dedicated story can follow if package install cost
  is significant.
