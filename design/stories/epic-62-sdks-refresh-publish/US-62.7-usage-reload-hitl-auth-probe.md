# US-62.7: All SDKs вЂ” Usage, Agent-Reload, Question/Permission, Auth Lifecycle, Probe

**Epic:** 62 - SDK Refresh, API-Surface Parity & Publish
**Priority:** Normal
**Depends on:** US-62.2, US-62.4, US-62.5 (base SDK structure must be in place)

## User Story

As an SDK consumer, I want SDK methods for usage/quota queries, explicit agent reloads, human-in-the-loop question/permission handling, the full auth lifecycle (register, logout, password reset, email verification), and anonymous model probing вЂ” so that I can build complete platform integrations without falling back to raw HTTP.

## Background

These route groups are live in the API server but no SDK exposes them. They span five functional areas added across Epics 12 (usage), 16 (HITL), 27a (agent reload), 49/54/56 (auth lifecycle), and the probe endpoint. Each is independently useful; this story bundles them because they are all small surface additions (1-5 methods each) and shipping them together avoids four separate cross-SDK PRs.

## Technical Details

### Auth register with Turnstile

The `register()` method signature should be:
```python
def register(self, email: str, password: str, username: str, *, turnstile_token: str = "") -> AuthResponse: ...
```
When Turnstile is enabled, the caller passes the token. The SDK sends it as a header or body field (verify which from the Turnstile middleware implementation). When Turnstile is disabled, the token is ignored by the server.

### Service group placement

- **Usage** в†’ new `UsageService` / `_UsageAPI` / `usage` group (standalone, not nested under workspaces)
- **Agent reload** в†’ on existing `Workspaces` service (it's a workspace-scoped operation)
- **Question/Permission** в†’ new `InputRequests` service or methods on `Sessions` (they are workspace-scoped but session-independent). Prefer a standalone group for clarity.
- **Auth lifecycle** в†’ extend existing `Auth` service (the `login`/`me`/`api-keys` group already exists)
- **DEK unlock** в†’ on `Auth` service (path is `/auth/unlock-dek`)
- **Probe** в†’ standalone `probeModels()` method on the client root or a small `Probe` group

### Question/Permission response shapes

The question/permission handlers are in `api/internal/handlers/proxy_input.go:28-85`. All five are pure proxies (`proxyToWorkspace`) вЂ” they validate the request ID format (regex `que_` or `per_` prefix), then forward the body to the workspace pod. The request body for reply/reject is opaque JSON passed through to the agent вЂ” the SDK should accept a generic body and forward it, not impose a typed shape.

The response shapes are opencode-proxied (the questions/permissions originate from the agent), so mark the types as `x-opencode-proxy` in the spec (US-62.1).
