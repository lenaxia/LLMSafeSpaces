# US-62.8: Publish Python/TypeScript/Java to Registries with Version Metadata

**Epic:** 62 - SDK Refresh, API-Surface Parity & Publish
**Priority:** High
**Depends on:** US-62.2, US-62.4, US-62.5 (parity-complete SDKs must exist before first publish)

## User Story

As an external developer integrating with LLMSafeSpaces, I want to install the SDKs from standard package registries (`pip install`, `npm install`, Maven) with importable version metadata, so that I can pin a specific version, detect breakage on upgrade, and build reproducibly.

## Background

None of the four SDKs are published to a registry. The external integrator's root-cause complaint was: "the SDK is stale, undocumented in its current state, and not published to a registry." Go modules are fetched directly from VCS (no registry publish needed), but Python, TypeScript, and Java all require registry publication for standard consumption.

This story publishes the parity-complete SDKs (post-US-62.2/62.4/62.5) as the first public release. The version number is coordinated with the OpenAPI spec version (US-62.1 bumps `info.version`).

## Acceptance Criteria

- [ ] **Python:** `pip install llmsafespaces` resolves from PyPI; `llmsafespaces.__version__` returns the published version
- [ ] **TypeScript:** `npm install @llmsafespaces/sdk` (or agreed name) resolves from npm; the package declares `version` in `package.json` matching the published version
- [ ] **Java:** the Maven artifact `com.llmsafespaces:sdk:<version>` resolves from Maven Central (or agreed registry)
- [ ] All three packages declare `AGPL-3.0-or-later` license (already present in manifests — verify)
- [ ] A GitHub Actions release workflow exists that publishes all three on tag push (e.g., `sdk/v1.1.0`)
- [ ] The release workflow uses registry API tokens stored as GitHub Secrets (PyPI API token, npm token, Maven Central credentials — including GPG signing for Maven)
- [ ] A `CHANGELOG.md` or release notes entry exists for the first release documenting the surface covered
- [ ] The version number is consistent across all four SDKs + the OpenAPI spec

## Technical Details

### Version coordination

All four SDKs + the OpenAPI spec publish at the same version. The version bump from `1.0.0` (current) to `1.1.0` (or `2.0.0` if the team considers the expanded surface a major change) is decided here and applied across:

| Artifact | Version field location |
|---|---|
| OpenAPI spec | `sdks/openapi.yaml` → `info.version` (US-62.1) |
| Python | `sdks/python/pyproject.toml:7` → `version` |
| TypeScript | `sdks/typescript/package.json:3` → `version` |
| Java | `sdks/java/pom.xml:8` → `<version>` |
| Go | `sdks/go/go.mod` (uses VCS tags, no explicit version field) |

### Release workflow (`.github/workflows/sdk-publish.yml`)

```yaml
name: Publish SDKs
on:
  push:
    tags: ['sdk/v*']

jobs:
  python:
    runs-on: ubuntu-latest
    environment: pypi
    permissions:
      id-token: write  # PyPI trusted publishing (OIDC)
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install build && cd sdks/python && python -m build
      - uses: pypa/gh-action-pypi-publish@release/v1
        # No password — trusted publishing via OIDC

  typescript:
    runs-on: ubuntu-latest
    environment: npm
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', registry-url: 'https://registry.npmjs.org' }
      - run: cd sdks/typescript && npm ci && npm run build
      - run: npm publish --access public
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}

  java:
    runs-on: ubuntu-latest
    environment: maven-central
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { java-version: '17', distribution: 'temurin' }
      # Maven Central requires GPG signing + Sonatype OSSRH staging
      # This is the most complex of the three — see Maven Central publishing guide
      - run: cd sdks/java && mvn deploy -P release
        env:
          MAVEN_USERNAME: ${{ secrets.MAVEN_USERNAME }}
          MAVEN_PASSWORD: ${{ secrets.MAVEN_PASSWORD }}
          MAVEN_GPG_PRIVATE_KEY: ${{ secrets.MAVEN_GPG_PRIVATE_KEY }}
          MAVEN_GPG_PASSPHRASE: ${{ secrets.MAVEN_GPG_PASSPHRASE }}
```

### Maven Central complexity

Maven Central publishing is the most involved:
1. Requires a Sonatype OSSRH account (or the newer Central Portal)
2. Requires a GPG key for artifact signing
3. Requires `pom.xml` metadata: `scm`, `developers`, `distributionManagement`, `license`
4. Uses a staging repository flow (upload → close → release)

If this is a blocker, the risk table mitigation applies: ship Python + TypeScript to their registries, ship Java as git-installable (`mvn install` from source), and defer Maven Central to a follow-up.

### Tag format

Use `sdk/v<version>` tags (e.g., `sdk/v1.1.0`) to distinguish SDK releases from platform releases. The workflow triggers only on this tag prefix.

## Design Reference

- `sdks/python/pyproject.toml` — Python build config (hatchling backend)
- `sdks/typescript/package.json` — npm package config
- `sdks/java/pom.xml` — Maven build config
- `sdks/go/go.mod` — Go module declaration
