# US-62.1: Refresh OpenAPI Spec to Match Current Router

**Epic:** 62 - SDK Refresh, API-Surface Parity & Publish
**Priority:** Critical
**Depends on:** —

## User Story

As an SDK maintainer and external integrator, I need the canonical OpenAPI specification to reflect every public route the API server actually serves, so that it remains the source of truth the README claims it is, and so that a future code-generation pipeline produces correct SDKs without inheriting drift.

## Background

`sdks/README.md:22` declares `sdks/openapi.yaml` "the single source of truth for the LLMSafeSpaces REST API contract." It is hand-written from `router.go` + `pkg/types/types.go` + handler implementations. The spec currently has 45 paths. The router serves ~70 public routes that are in scope for SDKs. The ~25-route gap accumulated across Epics 12 (usage), 16 (HITL), 27a (agent reload), 29 (models/env handler extraction), 30 (provider credentials), 41 (session queue), 43 (orgs), 49 (password reset / email verify), 54 (login discovery), and 55 (agent roles) — all shipped without updating the spec.

This story does NOT regenerate SDKs from the spec. It brings the spec current so the hand-written SDK parity stories (US-62.2/62.4/62.5) implement against an accurate contract, and so a future generation switch is possible.

## Acceptance Criteria

- [ ] Every public route in `api/internal/server/router.go` that is in scope for client SDKs (i.e., not in the out-of-scope list in the epic README) has a corresponding path entry in `sdks/openapi.yaml`
- [ ] A CI check exists that diffs `router.go` route registrations against spec paths and fails on mismatch. The check lives in the `sdks/validate/` tool (currently `main.go` — extend it) or as a separate script in `sdks/`.
- [ ] The CI check handles feature-gated routes: routes registered inside `if cfg.XxxHandler != nil` blocks are included in the spec with an `x-feature-gated: true` extension field, and the diff check knows to expect them.
- [ ] Request and response schemas for new paths reference existing component schemas where the types are shared (e.g., `Workspace`, `WorkspaceStatus`) and define new schemas only for genuinely new types.
- [ ] Proxy endpoints (those forwarding to opencode) are marked with `x-opencode-proxy: true` per the existing convention (`sdks/README.md:50`) — this includes the session queue routes and question/permission routes.
- [ ] `make validate` (the existing Go-based structural validator in `sdks/validate/main.go`) passes after all additions.
- [ ] The `info.version` field is bumped from `1.0.0` to reflect the expanded surface (the same version the SDKs will publish at — coordinate with US-62.8).
- [ ] No existing path or schema is removed or semantically changed — this is additive only.

**Restart** (router.go:1092):
- `POST /workspaces/{id}/restart` — was missing from spec despite being an early workspace lifecycle endpoint

### CI diff check

Add a route-coverage check to the existing `sdks/validate/` tool. The check maintains an explicit allowlist of expected route paths (the in-scope routes listed in this story's Technical Details section) and asserts that each one has a corresponding entry in `sdks/openapi.yaml`.

**Approach:** explicit path list in the validator source, NOT Go AST parsing of `router.go`. Rationale: AST parsing `router.go` for route registrations inside conditional blocks is fragile (Gin's router uses method calls on dynamic group variables), heavy to implement, and the route list is small enough (~70 paths) that an explicit list is more maintainable and auditable. If a new route is added to `router.go`, the CI check fails with "route X not in validator list" — a two-line fix to the list. This is simpler and more reliable than AST traversal.

The check runs as part of `make validate` so it gates every PR that touches the spec or router.

### What NOT to change

- Existing path schemas for `/workspaces`, `/sessions/{sessionId}/message`, `/secrets`, etc. — these are already correct and tested by the existing Hurl contract suite
- The OpenAPI version (3.0.3) — do not upgrade to 3.1
- The security schemes (`bearerAuth`, `apiKeyAuth`) — unchanged
