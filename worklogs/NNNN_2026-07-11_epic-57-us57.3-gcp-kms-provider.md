## Objective

Add GPCKMSProvider implementing RootKeyProvider using Google Cloud KMS symmetric Encrypt/Decrypt. Parallel to AWSKMSProvider — same threat model (KEK never leaves HSM), same prefix-sniffing dispatch, same fail-closed boot guard. Different SDK (gRPC vs HTTP), different auth (SA-JSON vs AWS credentials file).

## Key Decisions

- **Mock client interface vs fake gRPC server**: GCP KMS uses gRPC, not HTTP. Creating a fake gRPC server (bufconn + proto service implementation) is substantially more setup than the AWS httptest.Server approach. Using an interface mock (`gcpKMSClient`) is cleaner and follows the same pattern the AWS provider uses internally.
- **`WithAuthCredentialsFile` over `WithCredentialsFile`**: The latter is deprecated by staticcheck (SA1019) for security reasons. The former requires an explicit credential type (`option.ServiceAccount`), making the credential intent explicit.
- **No happy-path wiring test for GCP**: Unlike AWS (which lazy-loads credentials), GCP's SDK validates the SA JSON at `NewKeyManagementClient` time. A minimal fake JSON doesn't pass validation. The provider unit tests (mock client) + fail-closed wiring test provide adequate coverage without requiring real GCP credentials.
