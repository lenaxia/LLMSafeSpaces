# Worklog: v0.3.0 release pipeline

**Date:** 2026-07-11
**Session:** Build a real release pipeline: tag-triggered workflow, CHANGELOG-driven release notes, SBOM, Helm chart publishing, version surfacing.
**Status:** Complete

---

## Objective

Pre-this-PR, releases were git tags only. The CI workflow already builds and pushes multi-arch images to ghcr.io with semver tags on tag push, but:

1. The Helm chart was never packaged or published to a Helm repo — operators had to `git clone` and `helm install ./charts/llmsafespaces`.
2. No SBOM was generated — supply-chain consumers had nothing to feed into dependency-track or grype.
3. No GitHub Release was created — release notes lived only in PR titles.
4. The `-ldflags "-X main.version=..."` in `api/Dockerfile` and `controller/Dockerfile` was a no-op — `main.version` doesn't exist. The version was never actually compiled into the binary.
5. No `CHANGELOG.md` — release history was implicit in git log.

Goal: a real `v*.*.*`-tag-triggered pipeline that produces a complete release.

---

## Work Completed

### `pkg/version/version.go` (new)

Single source of truth for the build version string. Overridable at build time via:

    go build -ldflags "-X github.com/lenaxia/llmsafespaces/pkg/version.Version=v0.3.0"

Defaults to `"dev"` for local builds.

### `api/Dockerfile`, `controller/Dockerfile`

Fixed the existing (no-op) ldflags to actually inject into `pkg/version.Version` instead of the nonexistent `main.version`/`main.buildTime`/`main.commitSHA`. Removed the unused BUILD_TIME/COMMIT_SHA args from the controller Dockerfile (they were also no-ops).

### `api/internal/server/router.go`

`/livez` and `/health` now include the build version in the JSON response:

    {"status":"ok","version":"v0.3.0"}

Operators can verify which version is running via `curl /livez` without needing kubectl exec.

### `CHANGELOG.md` (new, seeded with v0.3.0)

[Keep a Changelog](https://keepachangelog.com/) format. v0.3.0 entry covers all 6 security PRs in the network hardening sweep + the Epic 57 US-57.1 KMS foundation + the Go 1.25.12 bump.

Each release entry links to the comparison view for that version range, and individual changes link to their PR.

### `.github/workflows/release.yml` (new)

Tag-triggered workflow with four jobs:

1. **`verify-changelog`** — refuses to release if CHANGELOG.md is missing a section for the tag's version. Fails loudly with an actionable error pointing at the missing heading.
2. **`wait-for-ci`** — waits for the CI workflow's `Lint`, `Test (full suite, race detector)`, and `Merge API Manifest` checks to complete on the tag's SHA. The CI workflow already does the image builds; this just gates on them succeeding.
3. **`publish-chart`** — uses `helm/chart-releaser-action` to package the chart as a .tgz and publish to the `gh-pages` branch's Helm repo. After this, `helm repo add llmsafespaces https://lenaxia.github.io/LLMSafeSpaces/` works.
4. **`generate-sbom`** — uses `anchore/sbom-action`'s syft to produce CycloneDX JSON for each of the 6 images (api, controller, base, frontend, relay-router, relay-proxy). Uploaded as build artifact.
5. **`create-release`** — extracts the CHANGELOG section for the version, prepends an image-reference table + install instructions, creates the GitHub Release, and attaches the SBOM artifacts.

### `.github/cr.yaml` (new)

chart-releaser config pointing at the `gh-pages` branch.

### `Makefile`

New targets:
- `release-verify-changelog VERSION=X.Y.Z` — pre-flight check.
- `release-tag VERSION=X.Y.Z` — validates semver shape, checks main is up to date with origin, verifies CHANGELOG has a section, refuses if the tag already exists, then cuts an annotated tag and pushes it.

### `charts/llmsafespaces/Chart.yaml`

Bumped `version` and `appVersion` from `0.2.2` to `0.3.0`. GitOps consumers (per the existing chart comment) read the chart version from Chart.yaml directly; the release workflow's `helm package --version` is a belt-and-braces override.

## Key Decisions

1. **Release notes come from CHANGELOG.md, not from auto-generated commit logs.** Commit messages are author-shaped; CHANGELOG entries are user-shaped. The release workflow fails if the section is missing — CHANGELOG is the source of truth for what shipped.
2. **No new image builds in the release workflow.** The CI workflow already does multi-arch builds with digest-merging and semver tagging on tag push. The release workflow waits for CI to finish and consumes what CI produced. This avoids duplicating ~600 lines of build matrix code.
3. **SBOM per image, not per release.** Each of the 6 images has different contents (different distroless bases, different dependencies). A single SBOM would lie about 5 of them. CycloneDX format because it's the SBOM standard grype/dependency-track consume.
4. **No code signing (cosign) in this PR.** Worth doing, but it adds: cosign install, OIDC keyless setup, a transparency log entry per image, a verification step. Separate PR — it's a meaningful chunk on its own and shouldn't block the v0.3.0 cut. Track as follow-up.
5. **`pkg/version.Version` is the canonical surface.** Every component that needs to report its version (`/livez`, future `--version` flag on CLIs) reads from here. No competing version strings.

## Assumptions stated and validated (Rule 7)

1. *CI workflow triggers on `tags: ['v*.*.*']` and pushes images with semver tags.* Validated by reading `.github/workflows/ci.yml:6` (trigger) and the `merge-api` job's `docker/metadata-action` tags config (lines 782-788).
2. *The existing `-X main.version=...` ldflags was a no-op.* Validated by `grep -rn "var version" api/cmd/api/main.go controller/cmd/*/main.go` — no such var exists.
3. *Helm `gh-pages` branch doesn't exist yet.* Validated by `git ls-remote origin gh-pages` — empty. The chart-releaser-action will create it on first release.
4. *`github.token` has sufficient permissions for `contents: write` (Release + gh-pages push) and `packages: write` (image pull for SBOM).* Validated by reading GitHub Actions token permission docs — both are within the default `GITHUB_TOKEN` scope when explicitly requested at workflow level.
