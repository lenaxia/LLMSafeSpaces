## Key Decisions

1. **v0.2.0 not v0.1.1.** Features (composer, Turnstile) since v0.1.0 require a minor bump. The user confirmed.
2. **Refresh the GitOps docs as part of this release, not a separate PR.** The existing test (`chart_gitops_doc_test.go`) hard-coded "0.1.0" as the version-pin signal. Bumping Chart.yaml without updating the test would fail CI. The minimum coherent change is: bump the version AND update the test + README to not reference the old literal. Doing it in two PRs would leave the repo in a broken state between them.
3. **Preserve the GitOps warning.** The temptation when bumping the chart version is to delete the `reconcileStrategy: Revision` warning ("we bump now, so ChartVersion works"). That's wrong — intermediate commits between releases (sha-/, ts-/, dev-tagged image builds for fast iteration) still don't bump Chart.yaml, so Flux's default strategy still misses them. The warning stays; only the rationale is updated.
4. **Tag the post-merge bot commit if it lands before tagging.** v0.1.0 was tagged on the repolint bot's worklog-renumber commit (`dd459855`), not on the feature commit. Following the same pattern keeps the release tied to the final state of main including worklog numbering.
