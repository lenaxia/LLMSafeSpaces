# Worklog: remove ZK framing + add docs maintenance runbook

**Date:** 2026-07-11
**Session:** Two related asks: (1) stop advertising "zero-knowledge secrets" since the SSO tier isn't actually ZK, and (2) document how the docs site gets maintained over time so the rot doesn't recur.
**Status:** Complete

---

## Objective

Two distinct problems:

1. **Inaccurate security framing.** The docs called the entire secret
   store "zero-knowledge" — but only the user-supplied secret path is
   actually ZK (cannot be decrypted without the user's password). The
   SSO client secrets, admin/org LLM credentials, and API-key DEKs are
   encrypted with the server KEK and the platform can always decrypt
   them. Advertising SSO while pretending it has the same security
   properties as password-protected secrets is dishonest.

2. **Drift risk.** The docs site is ~10,000 lines of hand-authored
   markdown. The AI reviewer on PR #527 caught two real drift bugs
   (controller flags wrong, helm-values selector keys abbreviated
   wrong). Without a maintenance runbook the rot will recur.

---

## Work Completed

### Removed all "zero-knowledge" references

Surveyed: 26 references across docs/, README.md, README-LLM.md,
pkg/README.md. Bulk-replaced via sed, then hand-tuned the load-bearing
pages.

Replacement framing uses **two-tier model** language where the
distinction matters:

- **User-password tier** — user-supplied secrets (LLM keys, SSH keys,
  env vars) encrypted with per-user DEKs derived from the password via
  Argon2id. Platform genuinely cannot decrypt without the password.
- **Server-KEK tier** — API keys, admin/org LLM credentials, SSO
  `client_secret`s encrypted under the master KEK. Platform needs to
  read these to function (e.g. completing an OIDC token exchange), so
  they are always decryptable by the platform.

The SSO tier being "less secure" than the password tier is an
intentional tradeoff (SSO requires the platform to present the client
secret to the IdP on the user's behalf). Documented explicitly in
`docs/operator/security.md` § "Two encryption tiers — read this if you
operate multi-tenant" so operators can reason about their own threat
model.

### Updated the framing pages

- `docs/index.md` — bullet point now distinguishes the two tiers
- `docs/architecture/secrets.md` — opening section, mermaid diagram
  labels, and the two-tier model section all use the new language

### Added docs-maintenance runbook

New page: `docs/contributing/docs-maintenance.md`. Covers:

- **Content inventory.** Every docs page categorised by drift risk
  (Reference = highest, Architecture = medium, Evergreen = low).
  For each page: the source-of-truth file to watch.
- **Maintenance triggers.** "When you change X, also update Y" table
  covering chart values, CLI flags, API routes, CRD fields, threat
  model, migrations, crypto, bootstrap scripts, releases.
- **Maintenance procedures** (A through F). Step-by-step for each
  common docs update: helm values, CLI flags, REST routes, threat
  model, changelog, local rebuild.
- **CI enforcement.** Documents what CI catches (broken links, missing
  nav entries) and what it doesn't (source-vs-docs drift on values,
  flags, routes). The latter is tracked as future work.
- **Audit cadence.** Per-PR (cheap, ~5–15 min per change), per-release
  (~1 hour end-to-end), quarterly (~2 hours). Cost is small if per-PR
  discipline holds; large if it doesn't.

Linked the runbook from the contributing index page's quick-links
table.

## Key Decisions

1. **Bulk sed + hand-tune, not rewrite.** The 26 ZK references were
   spread across many files. A bulk replacement ("zero-knowledge" →
   "encrypted") got most of them; hand-tuning fixed the awkward
   phrasings ("encrypted-at-rest store" → "encrypted secret store")
   and added the two-tier explanation where it mattered.
2. **No automation for drift prevention (yet).** The runbook documents
   what *could* be autogenerated (helm values, CRDs, CLI flags, REST
   routes) and explicitly tracks it as future work. Right now the
   right answer is discipline + CI link-checking + AI reviewer; if
   drift recurs despite that, we add generators.
3. **Two-tier model, not "weak/strong".** Calling the SSO tier "weak"
   would be misleading — it's encrypted at rest under a strong KEK,
   exactly like every other secrets-management system. The honest
   framing is "what the platform can decrypt vs what it cannot," and
   that's what the new table shows.
