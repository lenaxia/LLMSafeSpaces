## Objective

Add the `migrate-kek` CLI for zero-downtime cross-provider KEK migration. Walks the three KEK-protected Postgres tables, decrypts each row via a CompositeProvider (KMS-primary + static-fallback for legacy rows), and re-encrypts with the target KMS provider.

## Work Completed

### MigrationCoordinator (pkg/secrets/migration.go)

Parallel to RotationCoordinator (`rotation.go`). Key differences:
- Source = per-purpose CompositeProvider (handles both local and KMS prefix formats)
- Target = per-purpose KMS provider (encrypt only)
- `key_version` reset to 1 for migrated rows (D6)

Uses ported `purposeForTable` logic with both `provider_credentials` owner types and `api_keys/org_sso_configs` sharing `master-kek`.

### CLI (cmd/migrate-kek/main.go + store.go)

Supports `--kms aws|gcp` with per-purpose key flags. Same `--table`, `--resume-from`, `--dry-run`, `--redis-url`, `--db-url` as rotate-kek.

Store adapters follow rotate-kek's convention (PG/Redis stubs). Integration tests not feasible without live PG.

### P3 bug fix (review iteration)

The original implementation used a single-key `StaticKeyProvider` as the local fallback for all purposes. This couldn't decrypt legacy `api_keys` rows encrypted under the v1 `dek-cache`-derived key (the `master-kek` purpose needs both v1 and v2 keys). Fixed by constructing a `StaticKeyProviderMultiVersion` for the `master-kek` purpose with both v1=`dek-cache` and v2=`master-kek` keys. Regression test `MultiVersionFallback_DecryptsLegacyV1` pins this with a real multi-version provider.

### Key Decisions

- **Separate CLI from rotate-kek**: cross-provider migration has different inputs (KMS key ARNs/names vs master key files), a different deploy-both-providers step, and produces different output format (KMS-prefixed ciphertexts). Merging them in one CLI adds branches without sharing meaningful code.
- **PG/Redis stubs follow rotate-kek pattern**: the rotate-kek store.go has the same stubs. These CLIs are batch tools that SQL directly into Postgres; the connection wiring is deferred but structurally complete.
